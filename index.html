Introduction to SQL

⏱ 07:38 — 22:33

What is SQL?

SQL stands for Structured Query Language. It is the language you use to talk to a database. You write queries, send them to the database, and the database answers with data.

💬 Analogy: Think of data stored in different files like chaos. A database organises everything — and SQL is the language you speak to ask it questions.

Querying Data (SELECT)

⏱ 34:01 — 01:32:31

The Basic Query

A query is how you ask the database a question. It always starts with SELECT and uses FROM to tell SQL where to look.

⚙️ Execution order note: Even though you write SELECT first, SQL executes FROM first to fetch the data, then applies SELECT.

Select All Columns

SQLGet everything from a table
SELECT *
FROM customers;

Select Specific Columns

SQLChoose only the columns you need
SELECT
    first_name,
    country,
    score
FROM customers;
⚠️ No trailing comma! Never put a comma after the last column name — SQL will error expecting another column after it.

Comments in SQL

SQLTwo ways to write comments
-- This is a single-line comment

/* This is a
   multi-line comment */

SELECT * FROM customers; -- inline comment after code

WHERE — Filtering Rows

Use WHERE to filter rows based on a condition. Only rows that satisfy the condition appear in the results.

SQLSyntax
SELECT column1, column2
FROM table_name
WHERE condition;
SQLExamples
-- Customers with score not equal to 0
SELECT * FROM customers
WHERE score <> 0;

-- Customers from Germany (string values need single quotes)
SELECT first_name, country
FROM customers
WHERE country = 'Germany';
💡 String values need single quotes: 'Germany' ✅   Numbers do not: 500 ✅

ORDER BY — Sorting Results

Use ORDER BY to sort query results. Always specify the mechanism — don't rely on the default.

SQLSyntax
SELECT * FROM customers
ORDER BY score DESC;  -- highest first

SELECT * FROM customers
ORDER BY score ASC;   -- lowest first (default)

Nested Sorting (Multiple Columns)

Sort by a second column to break ties within the first.

SQLSort by country A–Z, then by highest score within each country
SELECT * FROM customers
ORDER BY
    country ASC,
    score   DESC;
📌 Column order matters! The first column in ORDER BY has the highest priority. Nested sorting only makes a difference when there are duplicate values in the first column.

GROUP BY — Aggregating Data

Combines rows with the same value and lets you aggregate them (sum, count, avg, etc.).

SQLTotal score per country
SELECT
    country,
    SUM(score) AS total_score,
    COUNT(id)  AS total_customers
FROM customers
GROUP BY country;
⚠️ Rule: Every column in SELECT must either be in GROUP BY or wrapped in an aggregate function. Mixing them causes an error.

Common Aggregate Functions

FunctionWhat It Does

COUNT(col)Number of rows (non-null)

MIN(col)Smallest value

HAVING — Filtering After Aggregation

Like WHERE, but for aggregated results. Always comes after GROUP BY.

SQLOnly show countries with avg score > 430 (excluding score = 0)
SELECT
    country,
    AVG(score) AS avg_score
FROM customers
WHERE score <> 0        -- filter BEFORE aggregation
GROUP BY country
HAVING AVG(score) > 430; -- filter AFTER aggregation
WHERE vs HAVING — the golden rule:
Use WHERE to filter before grouping (raw rows). Use HAVING to filter after grouping (aggregated results).

DISTINCT — Remove Duplicates

Placed right after SELECT. Forces each value to appear only once.

SQLUnique list of countries
SELECT DISTINCT country
FROM customers;
⚠️ Best practice: Don't blindly add DISTINCT to every query. It's an expensive operation. Only use it when your data actually has duplicates you want to remove.

TOP — Limit Results

Restricts how many rows are returned. Use with ORDER BY for meaningful top-N analysis.

SQLExamples
-- Get only 3 rows
SELECT TOP 3 * FROM customers;

-- Top 3 customers with highest score
SELECT TOP 3 *
FROM customers
ORDER BY score DESC;

-- 2 most recent orders
SELECT TOP 2 *
FROM orders
ORDER BY order_date DESC;
📌 Other databases use LIMIT n instead of TOP n. The logic is the same.

DDL Commands — Defining Structure

⏱ 01:32:31 — 01:43:44

DDL (Data Definition Language) commands define and modify the structure of your database. They don't return data rows — just a success or error message.

CREATE TABLE

Creates a new, empty table in your database.

SQLFull syntax with data types, constraints, and primary key
CREATE TABLE persons (
    id          INT           NOT NULL,
    person_name VARCHAR(50)  NOT NULL,
    birth_date  DATE          -- nullable: optional field
    phone       VARCHAR(15)  NOT NULL,
    CONSTRAINT PK_persons PRIMARY KEY (id)
);
💡 Recover a lost CREATE script: Right-click the table in SSMS → Script Table As → CREATE To → New Query Editor.

ALTER TABLE

Modifies an existing table — add or remove columns.

SQLAdd a column
ALTER TABLE persons
ADD email VARCHAR(50) NOT NULL;
SQLRemove a column
ALTER TABLE persons
DROP COLUMN phone;
⚠️ New columns are always added at the end of the table. To insert a column in the middle, you must drop and recreate the table from scratch.

DROP TABLE

Permanently deletes a table and all its data. Simple but dangerous.

SQL⚠️ Irreversible!
DROP TABLE persons;
🚨 Destroying a table is far easier than building one. Always double-check before running DROP.

INSERT — Adding New Rows

Method 1: Manual Values

SQLInsert one or multiple rows
INSERT INTO customers (id, first_name, country, score)
VALUES
    (6, 'Anna', 'USA', NULL),
    (7, 'Sam',  NULL, 100);

Method 2: Insert From Another Table

SQLMove data from source table to target table
INSERT INTO persons (id, person_name, birth_date, phone)
SELECT
    id,
    first_name,
    NULL          AS birth_date,  -- no birth date available
    'Unknown'     AS phone       -- default static value
FROM customers;
💡 Rules for INSERT: Number of columns must match number of values. You can skip nullable columns. You cannot skip NOT NULL columns (they need a value or a default).

UPDATE — Changing Existing Data

SQLAlways test your WHERE first before updating!
-- Step 1: Test the WHERE condition
SELECT * FROM customers WHERE id = 6;

-- Step 2: Run the update once confirmed
UPDATE customers
SET
    score   = 0,
    country = 'UK'
WHERE id = 6;
SQLUpdate multiple rows at once with a condition
-- Replace all NULL scores with 0
UPDATE customers
SET score = 0
WHERE score IS NULL;
🚨 Never run UPDATE without WHERE — it will update every single row in the table. Always test your WHERE condition with a SELECT first.

DELETE — Removing Rows

SQLDelete specific rows
-- Test first
SELECT * FROM customers WHERE id > 5;

-- Then delete
DELETE FROM customers
WHERE id > 5;

SQLDelete ALL rows — much faster than DELETE for large tables
TRUNCATE TABLE persons;

BETWEEN — Range Operator

Checks if a value falls within a range. Both boundaries are inclusive.

SQLTwo equivalent ways to write a range

WHERE score BETWEEN 100 AND 500;

-- Equivalent using comparison operators (more explicit — preferred)
SELECT * FROM customers
WHERE score >= 100
  AND score <= 500;
💡 The instructor prefers the comparison operator approach — it's more explicit about whether boundaries are inclusive or exclusive.

IN / NOT IN — Membership Operators

Check whether a value belongs to a list. Cleaner than chaining multiple OR conditions.

SQLIN vs chained OR — same result, IN is cleaner
-- Verbose way (avoid)
SELECT * FROM customers
WHERE country = 'Germany' OR country = 'USA';

-- Clean way with IN (preferred)
SELECT * FROM customers
WHERE country IN ('Germany', 'USA');

-- NOT IN — exclude from list
SELECT * FROM customers
WHERE country NOT IN ('Germany', 'USA');
💡 If you find yourself repeating column = val1 OR column = val2 OR ..., switch to IN. It's shorter, easier to extend, and has better performance.

SQLPattern examples

SELECT * FROM customers WHERE first_name LIKE 'M%';

-- Names ending with N
SELECT * FROM customers WHERE first_name LIKE '%N';

-- Names containing R anywhere
SELECT * FROM customers WHERE first_name LIKE '%R%';

-- R is exactly at the 3rd character position
SELECT * FROM customers WHERE first_name LIKE '__R%';
📌 % is far more commonly used than _. The %text% pattern (contains) is the most popular pattern for searching inside values.

SQL JOINs

⏱ 2:50:46 — 4:02:09

Why Use JOINs?

Use CaseDescriptionExample
Recombine dataData about one topic is spread across multiple tables — join them to see the big picturecustomers + addresses + orders + reviews
Data enrichmentAdd extra info from a reference/lookup table to your main tableAdd zip codes from a country reference table

JOIN Syntax Template

SQLAlways specify the join type — don't rely on defaults
SELECT
    c.id,
    c.first_name,
    o.order_id,
    o.sales
FROM customers   AS c
INNER JOIN orders AS o
    ON c.id = o.customer_id;
💡 Best Practices: Always use table aliases (c, o). Always prefix columns with their alias (c.id). Always specify the join type explicitly — never rely on the default.

INNER JOIN

Returns only the rows where there is a match in both tables. Non-matching rows from either table are excluded.

SQLOnly customers who placed at least one order
SELECT
    c.id,
    c.first_name,
    o.order_id,
    o.sales
FROM customers AS c
INNER JOIN orders AS o
    ON c.id = o.customer_id;
Order doesn't matter for INNER JOIN — FROM A INNER JOIN B gives the same result as FROM B INNER JOIN A.

Use for: combining data from multiple tables (big picture) or filtering rows by existence in another table.

LEFT JOIN

Returns all rows from the left table and only matching rows from the right. Where there's no match, right-side columns show NULL.

SQLAll customers including those without orders
SELECT
    c.id,
    c.first_name,
    o.order_id,
    o.sales
FROM customers AS c          -- LEFT table = main/primary source
LEFT JOIN orders AS o         -- RIGHT table = secondary/enrichment
    ON c.id = o.customer_id;
-- Customers without orders will show NULL in order_id and sales
⚠️ Order matters! The table in FROM is the LEFT table. The table after LEFT JOIN is the RIGHT. Swapping them gives different results.

Use for: when the left table is your primary source and you don't want to lose any of its rows. The most commonly used join type.

RIGHT JOIN

The mirror of LEFT JOIN — returns all rows from the right table, only matching from the left. NULLs appear on the left where there's no match.

SQLAll orders including those without a matching customer
SELECT
    c.id,
    c.first_name,
    o.order_id,
    o.sales
FROM customers AS c
RIGHT JOIN orders AS o
    ON c.id = o.customer_id;
🏆 Pro tip (instructor's preference): Skip RIGHT JOIN entirely. Just swap the table order and use LEFT JOIN instead — you get the same result and your queries stay consistent and easier to read.
SQLRIGHT JOIN → rewritten as LEFT JOIN (preferred)
-- Instead of: FROM customers RIGHT JOIN orders
-- Write this:  FROM orders LEFT JOIN customers

SELECT
    c.id,
    c.first_name,
    o.order_id,
    o.sales
FROM orders AS o             -- orders is now the LEFT (primary) table
LEFT JOIN customers AS c
    ON c.id = o.customer_id;

FULL JOIN (FULL OUTER JOIN)

Returns everything from both tables. Matching rows are combined; non-matching rows from either side show NULL in the missing columns.

SQLAll customers AND all orders, matched where possible
SELECT
    c.id,
    c.first_name,
    o.order_id,
    o.sales
FROM customers AS c
FULL JOIN orders AS o
    ON c.id = o.customer_id;
Order doesn't matter for FULL JOIN — both tables are equally important. Use when you don't want to miss a single row from either side.

Advanced JOINs — Anti-Joins

⏱ 3:27:29 — 4:02:09

Anti-joins find rows in one table that have no match in the other. They don't have their own keyword — you build them by combining a classical join with a WHERE ... IS NULL filter.

Left Anti-Join

Rows in the left table that have no match in the right table.

SQLCustomers who have never placed an order
SELECT c.*
FROM customers AS c
LEFT JOIN orders AS o
    ON c.id = o.customer_id
WHERE o.customer_id IS NULL;  -- NULL means no match found

Logic: Step 1 — do a LEFT JOIN (you'll see NULLs on the right for unmatched rows). Step 2 — filter WHERE the right-side key IS NULL.

Right Anti-Join

Rows in the right table that have no match in the left. (Again, best rewritten as a left anti-join by swapping tables.)

SQLOrders with no valid customer — rewritten as LEFT anti-join (preferred)
SELECT o.*
FROM orders AS o              -- orders moved to LEFT position
LEFT JOIN customers AS c
    ON c.id = o.customer_id
WHERE c.id IS NULL;           -- no matching customer exists

Full Anti-Join

All unmatched rows from both tables — the opposite of INNER JOIN.

SQLCustomers without orders AND orders without customers
SELECT c.id, c.first_name, o.order_id, o.sales
FROM customers AS c
FULL JOIN orders AS o
    ON c.id = o.customer_id
WHERE o.customer_id IS NULL   -- customer has no order
   OR c.id IS NULL;            -- order has no customer

Use OR (not AND) — you want rows where either side is unmatched.

CROSS JOIN

Every row from the left combined with every row from the right — all possible combinations. No join condition needed. Rows = left_count × right_count.

SQL5 customers × 4 orders = 20 rows
SELECT *
FROM customers
CROSS JOIN orders;  -- no ON condition needed
⚠️ Use with caution. On large tables, cross joins generate enormous result sets and will hammer your database. Useful for generating test data or enumerating all combinations (e.g. all products × all colours).

JOIN Decision Tree

How to choose the right join type every time:

Want only MATCHING rows?
→ INNER JOIN
Want ALL rows, one table is "main"?
→ LEFT JOIN (start from the main table)
Want ALL rows from BOTH tables?
→ FULL JOIN (order doesn't matter)
Want UNMATCHED rows from one side?
→ LEFT JOIN + WHERE right_key IS NULL
Want UNMATCHED rows from BOTH sides?
→ FULL JOIN + WHERE either key IS NULL
Want ALL combinations?
→ CROSS JOIN (use carefully!)

The instructor's personal practice: always start from the main table and chain LEFT JOINs for everything else, then use WHERE to control which rows appear in the final result.

Joining Multiple Tables

⏱ 3:51:46 — 4:02:09

Real projects often need 4+ tables joined together. The pattern is always the same: one main table, everything else LEFT JOINed to it using the appropriate key from the ER diagram.

SQLOrders with customer, product, and employee details — 4 tables
SELECT
    o.OrderID,
    o.Sales,
    c.FirstName     AS CustomerFirstName,
    c.LastName      AS CustomerLastName,
    p.Product       AS ProductName,
    p.Price,
    e.FirstName     AS EmployeeFirstName,
    e.LastName      AS EmployeeLastName
FROM sales.orders   AS o         -- main table: orders
LEFT JOIN sales.customers AS c
    ON o.CustomerID = c.CustomerID
LEFT JOIN sales.products AS p
    ON o.ProductID = p.ProductID
LEFT JOIN sales.employees AS e
    ON o.SalesPersonID = e.EmployeeID;
💡 Best Practices for multi-table joins:
  • Always join from the main/master table outward — don't chain secondary tables to each other.
  • Always use table aliases — queries get unreadable fast without them.
  • When two tables share a column name (e.g. both have FirstName), use aliases: c.FirstName AS CustomerFirstName.
  • Use your ER diagram to identify the correct join keys.
  • Double-check join keys — wrong keys produce silently wrong results with no error message.

Set Operators — Combining Rows

⏱ 4:02:09 — 4:47:41

Set operators stack the rows of two or more queries into one result. Both queries must select the same number of columns.

UNION
All rows from both queries combined
Removes duplicates ✓
UNION ALL
All rows from both queries combined
Keeps duplicates · Faster ⚡
EXCEPT
Rows in query 1 that are NOT in query 2
Order matters! Removes duplicates ✓
INTERSECT
Rows that appear in BOTH queries
Removes duplicates ✓

The 5 Rules of Set Operators

1
ORDER BY once, at the end. You cannot use ORDER BY inside individual queries — only once at the very end of the entire combined query.
2
Same number of columns. Each query must SELECT the same count of columns. Mismatches cause an error.
3
Matching data types. The 1st column of query 1 must be the same data type as the 1st column of query 2, and so on.
4
Same column order. SQL maps columns positionally (1st to 1st, 2nd to 2nd) — not by name. Wrong order = wrong data, no error.
5
First query controls output names. Column aliases and names in the result come from the first query only. Aliases in other queries are ignored.
🚨 Silent data corruption risk: SQL only validates data types and counts — not meaning. If you accidentally swap first_name and last_name in one query, SQL will produce results without any error. Always double-check column mappings.

UNION

Combines all rows from both queries and automatically removes duplicates.

SQLCombine customers and employees into one people list (no duplicates)
SELECT first_name, last_name FROM sales.customers
UNION
SELECT first_name, last_name FROM sales.employees
ORDER BY first_name ASC;  -- ORDER BY only at the end

UNION ALL

Same as UNION but keeps all duplicates. Significantly faster than UNION because it skips the deduplication step.

SQLInclude duplicates — use when you know there are none, or want to see them
SELECT first_name, last_name FROM sales.customers
UNION ALL
SELECT first_name, last_name FROM sales.employees;
🏆 Performance tip: If you know your data has no duplicates across queries, always use UNION ALL. It's faster because it skips the deduplication pass.

EXCEPT

Returns rows from the first query that do not exist in the second query. Think: "what's in A but not in B?"

⚠️ Order matters! A EXCEPT B ≠ B EXCEPT A. The first query defines what you're looking at; the second is used as a filter.
SQLEmployees who are NOT customers
-- Employees first because we want to find employees
SELECT first_name, last_name FROM sales.employees
EXCEPT
SELECT first_name, last_name FROM sales.customers;

INTERSECT

Returns only the rows that appear in both queries — the common data.

SQLPeople who are both customers AND employees
SELECT first_name, last_name FROM sales.employees
INTERSECT
SELECT first_name, last_name FROM sales.customers;

Best Practice: Add a Source Column When Combining Tables

SQLTag each row with where it came from
SELECT 'orders'         AS source_table, order_id, product_id, sales
FROM sales.orders
UNION ALL

💡 Avoid SELECT * with set operators — list columns explicitly. If one table's schema changes (columns reordered, renamed, added), a SELECT * will silently map wrong columns and corrupt your results.

PARTITION BY — Dividing Into Windows

Works like GROUP BY but doesn't collapse rows. Divides the dataset into sub-groups (windows) for the calculation.

SQLThree levels of window scope
SELECT
    order_id,
    order_date,
    product_id,
    sales,

    -- Level 1: whole table = one window (no PARTITION BY)

    SUM(sales) OVER(PARTITION BY product_id, order_status) AS sales_by_product_status
FROM sales.orders;

ORDER BY Inside the Window

Sorts data within each partition. Required for ranking and value functions. For aggregate functions, adding ORDER BY activates the default frame (running total effect).

⚠️ When you use ORDER BY inside a window aggregate, SQL activates a hidden default frame: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. This turns a simple sum into a running total. Always be aware of this.

Frame Clause — Defining the Subset

Restricts which rows within a window are included in the calculation. Only matters when ORDER BY is used.

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  ← default when ORDER BY used (running total)
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING  ← entire window always (same as no frame)
ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWING  ← needed for LAST_VALUE to work correctly
ROWS BETWEEN 2 PRECEDING AND CURRENT ROW  ← rolling 3-row window
ROWS BETWEEN CURRENT ROW AND 1 FOLLOWING  ← current + next row

Rules & Limitations

  • Window functions can only appear in SELECT and ORDER BY clauses — not in WHERE, GROUP BY, or HAVING.
  • You cannot nest window functions inside each other.
  • WHERE filters are applied before window functions calculate.
  • Window functions can be used alongside GROUP BY — but only if you reference columns/expressions that are already in the GROUP BY or are aggregates of it.

SELECT

FROM sales.orders

Common Table Expressions (CTEs)

⏱ 14:18:08 — 15:35:02

A CTE is a temporary named result set — like a virtual table that lives inside one query. Defined with the WITH clause. The result is stored in the cache during query execution and destroyed once done.

CTE vs Subquery: A subquery result can only be used once, in one place. A CTE result can be used multiple times across the same query. Think of a CTE as a named table that exists only inside your query.
CTE vs View: A CTE is temporary (one query scope, lives in cache). A View is permanent (persisted in the database, globally accessible). Use a CTE when the logic only matters in this one query. Use a View when the logic matters across many queries.

Standalone CTE — Basic Syntax

SQLSingle CTE — total sales per customer
WITH cte_total_sales AS (

        customer_id,
        SUM(sales) AS total_sales
    FROM sales.orders
    GROUP BY customer_id
)

    c.customer_id,
    c.first_name,
    c.last_name,
    ct.total_sales
FROM sales.customers c
LEFT JOIN cte_total_sales ct ON c.customer_id = ct.customer_id
ORDER BY c.customer_id;
SQLMultiple standalone CTEs — building a customer report
WITH
cte_total_sales AS (
    SELECT customer_id, SUM(sales) AS total_sales
    FROM sales.orders GROUP BY customer_id
),
-- Comma (not WITH) for each additional CTE
cte_last_order AS (
    SELECT customer_id, MAX(order_date) AS last_order
    FROM sales.orders GROUP BY customer_id
),
cte_customer_rank AS (
    SELECT
        customer_id, total_sales,
        RANK() OVER(ORDER BY total_sales DESC) AS sales_rank
    FROM cte_total_sales   -- ← nested: this CTE uses the first one
)

    c.first_name, c.last_name,
    ts.total_sales,

LEFT JOIN cte_last_order   lo ON c.customer_id = lo.customer_id
LEFT JOIN cte_customer_rank cr ON c.customer_id = cr.customer_id;

Recursive CTEs

Two parts connected by UNION ALL: the anchor query (runs once, gives the starting point) and the recursive query (runs repeatedly, references the CTE itself, must include a stopping condition).

SQLGenerate a sequence of numbers 1–20
WITH series AS (
    -- Anchor query: starting point
    SELECT 1 AS my_number

    UNION ALL

    -- Recursive query: each iteration adds 1
    SELECT my_number + 1
    FROM series
    WHERE my_number < 20   -- stopping condition (not ≤ 20 — that adds 21)
)
SELECT * FROM series
OPTION (MAXRECURSION 100);  -- default limit; raise if needed
SQLEmployee hierarchy — find each employee's level
WITH employee_hierarchy AS (
    -- Anchor: the top-level manager (no manager above them)
    SELECT
        employee_id,
        first_name,
        manager_id,
        1 AS level
    FROM sales.employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive: find employees whose manager is in the current result
    SELECT
        e.employee_id,
        e.first_name,
        e.manager_id,
        ch.level + 1   -- increment level
    FROM sales.employees e
    INNER JOIN employee_hierarchy ch ON e.manager_id = ch.employee_id
    -- INNER JOIN acts as the stopping condition (no more matches = stop)
)
SELECT * FROM employee_hierarchy;

Indexes

⏱ 18:23:30 — 20:20:00

An index is a data structure that provides fast access to rows without scanning every page. Like a book index — jump straight to the right page rather than reading everything.

🚨 Golden rule: Avoid overindexing. More indexes = slower writes (every INSERT/UPDATE/DELETE must update all indexes) + confused execution plans. Less is more. A few well-chosen indexes beat many scattered ones.

Heap (No Index)

Data inserted in any order with no structure. Very fast writes. Very slow reads — requires a full table scan (every page, every row). Use for staging/temp tables only.

Clustered Index — B-Tree

Physically sorts and stores the data rows in order. The leaf nodes of the B-tree are the data pages. Only one allowed per table. Best for primary keys.

SQLCreate, drop, and check clustered index
-- Create
CREATE CLUSTERED INDEX idx_db_customers_customer_id
ON sales.db_customers (customer_id);

-- Drop
DROP INDEX idx_db_customers_customer_id
ON sales.db_customers;

-- View all indexes on a table (system SP)
EXEC sp_helpindex 'sales.db_customers';

Table Partitioning

Divides one large table into multiple smaller partitions (by year, region, etc). To queries and users it still looks like one table — but SQL only scans the relevant partition. Key benefit: partition elimination.

4-Step Setup Process

-- Creates 4 partitions: ≤2023 | 2024 | 2025 | ≥2026

-- Step 2: File GROUPS (logical containers)

ALTER DATABASE sales_db ADD FILEGROUP FG_2024;
ALTER DATABASE sales_db ADD FILEGROUP FG_2025;
ALTER DATABASE sales_db ADD FILEGROUP FG_2026;

AS PARTITION partition_by_year

20+ Performance Best Practices

⏱ 21:43:44 — 22:24:30
📐 Golden rule: Always test with the execution plan. If a rewrite shows no improvement in the plan for your table size, keep the simpler, more readable version. Optimise only when needed.

Query Writing

01
Avoid SELECT *
Always list only the columns you need. SELECT * reads and transmits unnecessary data on every query.
BAD: SELECT * FROM orders
GOOD: SELECT order_id, sales FROM orders
02
Avoid Unnecessary DISTINCT & ORDER BY
Both are expensive operations. Only use if actually needed — don't add them as habit.
03
Limit rows when exploring
Use TOP 10 when peeking at large tables. Don't fetch 100M rows to see what a table looks like.
04
No functions on WHERE columns
Functions on indexed columns prevent index use. Rewrite to keep the column bare.
BAD: WHERE YEAR(order_date) = 2025
GOOD: WHERE order_date BETWEEN '2025-01-01' AND '2025-12-31'
05
No leading wildcards in LIKE
LIKE '%gold%' can't use an index. LIKE 'gold%' can.
06
Use IN instead of multiple OR
IN is cleaner and performs better than chained OR conditions.
BAD: customer_id=1 OR customer_id=2
GOOD: customer_id IN (1, 2, 3)

Joining Tables

07
Prefer INNER JOIN
INNER JOIN processes only matching rows → fastest. LEFT/RIGHT process more rows. OUTER is slowest.
08
Use ANSI JOIN syntax
Always use explicit JOIN ... ON. Avoid the implicit comma-separated syntax — harder to read and optimise.
09
Index JOIN columns (ON clause)
Create nonclustered indexes on foreign-key columns used in ON conditions to avoid full scans.
10
Filter/aggregate before joining (big tables)
Prepare and reduce data in a CTE or subquery first, then join. Fewer rows to join = faster.
11
Avoid OR in JOIN conditions
OR in ON disables index usage. Replace with UNION ALL of two separate joins.
12
EXISTS over IN for existence checks
When filtering one table by rows in another, EXISTS often outperforms IN (stops at first match). Avoid correlated subqueries — worst performance.

Aggregations & Unions

13
Column store for large fact tables
Apply clustered column store index to fact tables in data warehouses. Aggregations become dramatically faster.
14
Pre-aggregate + store for reporting
If a complex query takes minutes, insert its result into a table at night. Reports then hit the fast table, not the complex query.
15
UNION ALL over UNION
Use UNION ALL when duplicates are acceptable or impossible. UNION checks every row for duplicates — expensive.
16
Avoid redundant logic
If you're scanning the same table 4× in one query, rewrite using a window function or CTE. Don't repeat logic.

DDL Design

17
Avoid VARCHAR(MAX) / TEXT
Use appropriate types (INT, DATE, FLOAT). VARCHAR/TEXT kills index performance. Define realistic lengths.
18
Use NOT NULL constraints
Helps the optimizer, reduces index size, and enforces data quality.
19
Every table needs a clustered PK
SQL Server creates a clustered index on the PK automatically. Without a PK → heap structure → slow reads.
20
Index FK columns (if used in joins)
Foreign key columns frequently appear in JOIN ON conditions. Create a nonclustered index on them.

What Is a Data Warehouse?

Bill Inmon's Definition
A data warehouse is a subject-oriented, integrated, time-variant, and non-volatile collection of data designed to support the management's decision-making process.
PropertyMeaning
Subject-OrientedFocuses on a specific business area — sales, customers, finance
IntegratedMerges multiple source systems into one consistent model
Time-VariantStores historical data — you can travel back in time
Non-VolatileOnce data enters, it is never deleted or modified

Why Do Companies Need a Data Warehouse?

Without one: each analyst manually extracts, transforms, and shares data via Excel / PowerPoint / Power BI — slow, error-prone, inconsistent staleness (reports 40 days vs 5 days old), impossible to merge sources.

With one: ETL automates extraction and transformation; a single source of truth feeds all reports; data is integrated across sources; history is preserved; the process runs in hours or minutes.

Restaurant Analogy
The data warehouse is like a professional kitchen. Suppliers (source systems) deliver raw ingredients daily. The kitchen cleans, organises, and stores them properly. When an order comes in (report request), a perfect dish is served instantly — instead of each customer waiting while a chef goes shopping.

Roles

  • Data Engineer: builds the ETL pipelines and the warehouse database
  • Data Analyst: consumes the warehouse, builds models, reports to stakeholders

The Secret: Dimensions vs Measures

The Framework
If a column is numeric AND makes sense to aggregate → it's a Measure. Otherwise → it's a Dimension.
Dimension (GROUP BY this)
Non-numeric columns
Numeric but aggregation is meaningless (e.g. customer ID)
Answers: who, what, where, when
country category gender product_name customer_id order_date
Measure (AGGREGATE this)
Numeric AND aggregation is meaningful
Derived numeric (e.g. age from birth_date)
Answers: how much, how many
sales_amount quantity price age lifespan_months
